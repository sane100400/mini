AI 부업을 프리랜서 일로 만들고 싶다면 "AI를 잘 씁니다"라는 말만으로는 부족하다. 의뢰인은 어떤 도구를 쓰는지보다 무엇을 만들어봤는지를 본다. 특히 재택 프리랜서는 직접 만나는 일이 적기 때문에 클릭해서 확인하는 결과물이 필요하다.

처음부터 거창한 웹사이트를 만들 필요는 없다. 블로그 글 한 편, 숏폼 대본 한 묶음, 상세페이지 문구 전후 비교 하나면 시작할 수 있다. 작아도 완성되어 있고, 작업 범위가 분명하면 포트폴리오 역할을 한다.

| 결과물 | 보여줄 능력 | 공개 방식 |
| --- | --- | --- |
| 블로그 글 | 질문을 좁히고 근거를 붙이는 능력 | 블로그 링크 |
| 숏폼 대본 | 짧은 문장과 흐름을 만드는 능력 | 문서, 이미지, PDF |
| 상세페이지 문구 | 고객 질문을 판매 문장으로 바꾸는 능력 | 전후 비교표 |
| 랜딩페이지 | 화면을 끝까지 배포하는 능력 | GitHub, Pages 링크 |

## 첫 번째 샘플은 블로그 글이다

블로그 글은 AI 부업 포트폴리오의 기본이다. 글을 잘 쓴다는 말보다 하나의 질문을 잡고, 근거를 붙이고, 결론까지 밀고 가는 힘을 보여준다.

예를 들어 "AI 데이터 라벨링 부업 팁, Outlier·DataAnnotation 한국어 작업 조건 확인" 같은 글은 단순 소개가 아니다. 플랫폼 공식 화면을 보고, 한국인이 확인해야 할 조건을 정리하고, 어떤 사람에게 맞는지 판단한다. 이 과정이 그대로 포트폴리오가 된다.

| 블로그 샘플에 넣을 것 | 이유 |
| --- | --- |
| 제목을 좁힌 과정 | 기획 능력을 보여줌 |
| 공식 화면이나 캡처 | 실제 확인했다는 근거 |
| 표나 체크리스트 | 읽는 사람이 빠르게 판단 |
| 참고 링크 | 출처와 신뢰 |
| 다음 글 연결 | 블로그 운영 감각 |

좋은 블로그 샘플은 검색자가 실제로 묻는 질문에서 시작한다. "AI 부업 추천"처럼 넓은 글보다 "AI 숏폼 대본을 외주 상품으로 만들 때 수정 범위를 어디까지 잡을까"처럼 좁은 글이 낫다.

## 두 번째 샘플은 숏폼 대본이다

숏폼 대본은 짧지만 포트폴리오로 쓰기 좋다. 첫 3초 후킹, 본문 흐름, 마지막 행동 유도가 분명하게 보이기 때문이다. 영상 편집까지 하지 않아도 대본만으로 작업자의 감각을 보여줄 수 있다.

| 대본 샘플 구성 | 예시 |
| --- | --- |
| 대상 | 전자책 판매자 |
| 목표 | 블로그 글로 유입 |
| 길이 | 20초 |
| 첫 화면 문구 | AI 부업 글감이 매번 막힌다면 |
| 내레이션 | 키워드보다 먼저 막히는 지점을 적어야 합니다 |
| CTA | 체크리스트는 블로그에 정리해뒀습니다 |

AI를 쓴 대본은 다듬은 흔적이 중요하다. AI 초안은 길고 안전한 표현을 많이 쓴다. 포트폴리오에는 한두 문장만 전후 비교로 보여줘도 된다.

| AI 초안 느낌 | 다듬은 문장 |
| --- | --- |
| 효율적인 방법을 통해 생산성을 높일 수 있습니다 | 하루 20분짜리 루틴부터 만듭니다 |
| 다양한 전략을 활용할 수 있습니다 | 제목 3개를 먼저 써보고 하나만 고릅니다 |
| 도움이 되는 콘텐츠를 제공합니다 | 바로 쓸 수 있는 체크리스트를 남깁니다 |

## 세 번째 샘플은 상세페이지 문구다

상세페이지 문구는 판매자에게 바로 설명하기 좋은 결과물이다. 상품 사진은 있는데 문구가 약한 판매자가 많다. 이때 AI를 써서 고객 질문을 뽑고, 문구를 짧게 고치고, 전후 비교를 보여준다.

| 수정 전 | 수정 후 |
| --- | --- |
| 좋은 소재로 만든 편안한 제품입니다 | 안쪽 면을 부드럽게 처리해 오래 착용해도 덜 거슬립니다 |
| 실용적인 텀블러입니다 | 350ml 용량이라 출근 가방에 넣기 부담이 적습니다 |
| 프리미엄 강의를 제공합니다 | 첫 주에는 과제 제출 흐름부터 같이 잡습니다 |

완전히 새로운 사실을 만들면 안 된다. 기존 정보 안에서 고객이 궁금해할 부분을 앞쪽으로 꺼내는 것이 포인트다. 이 결과물은 "AI 초안에서 지워야 할 말", "상세페이지 문구는 고객 질문에서 시작한다" 같은 블로그 글로도 연결된다.

## 공개 링크로 신뢰를 만든다

포트폴리오는 파일만 모아두면 전달이 어렵다. 공개 링크가 있으면 훨씬 편하다. 상단 대표 이미지는 GitHub Pages 문서를 직접 캡처한 화면이다. 개발자만 쓰는 공간처럼 보이기도 하지만, 바이브 코딩 결과물이나 간단한 웹 샘플을 보여줄 때 유용하다.

| 공개 위치 | 어울리는 결과물 |
| --- | --- |
| 블로그 | 원고 샘플, 작업 기준, 후기형 글 |
| GitHub | 바이브 코딩 샘플, README, 코드 |
| GitHub Pages | 랜딩페이지, 계산기, 포트폴리오 |
| PDF/이미지 | 숏폼 대본, 상세페이지 전후 비교 |

README도 포트폴리오의 일부다. 무엇을 만들었는지, 어떤 도구를 썼는지, 어떤 문제를 해결했는지, 어디까지가 샘플인지 적어야 한다. "AI로 만들었습니다"만 쓰면 약하다. "v0로 초안을 만들고 Cursor로 문구와 반응형 레이아웃을 수정한 뒤 GitHub Pages로 배포했다"처럼 작업 과정을 적으면 신뢰가 생긴다.

## 처음 만들 포트폴리오 3개

| 순서 | 결과물 | 완료 기준 |
| --- | --- | --- |
| 1 | 블로그 글 1편 | 공식 화면, 표, 결론 포함 |
| 2 | 숏폼 대본 5개 | 업종, 길이, CTA 표시 |
| 3 | 상세페이지 문구 전후 비교 | 수정 전, 문제점, 수정 후 표시 |

포트폴리오 페이지는 복잡할 필요가 없다. 자기소개 한 단락, 가능한 일 세 가지, 샘플 3개, 작업 범위, 연락 방법이면 충분하다. 말보다 결과물이 먼저다. 재택 프리랜서에게 신뢰는 클릭해서 확인하는 증거에서 나온다.

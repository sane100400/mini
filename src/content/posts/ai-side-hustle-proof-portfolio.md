AI 부업을 프리랜서 일로 바꾸려면 포트폴리오가 필요하다. 문제는 많은 사람이 포트폴리오를 거창하게 생각한다는 점이다. 처음부터 홈페이지를 만들고, 긴 소개서를 쓰고, 모든 서비스를 한 번에 보여주려고 한다.

초반에는 결과물 3개면 충분하다. 블로그 글 1개, 숏폼 대본 1개, 상세페이지 문구 1개다. 이 세 가지는 AI 부업에서 가장 많이 연결되는 작업이고, 의뢰인도 결과를 바로 이해한다.

![콘텐츠 작업용 책상과 노트북](./images/content-desk.jpg)

## 첫 번째 결과물은 블로그 글

블로그 글 샘플은 1,200자 안팎이면 된다. 중요한 것은 문장보다 구조다. 제목, 소제목, 고객 질문, 마무리 행동이 보여야 한다. 예쁜 말보다 읽는 사람이 다음 행동을 알 수 있어야 한다.

샘플 주제는 실제 업종으로 잡는다. "동네 필라테스 상담 글", "수제 간식 쇼핑몰 안내 글", "중고 노트북 매입 안내 글"처럼 흔한 업종이 좋다. 의뢰인이 자기 일과 비교하기 쉽기 때문이다.

## 두 번째 결과물은 숏폼 대본

숏폼 대본은 30초짜리로 만든다. 긴 영상보다 짧은 영상이 샘플로 좋다. 첫 3초 문장, 화면 자막, 말하는 문장, 마지막 행동 유도가 나뉘어 있으면 된다.

![스마트폰으로 짧은 영상을 촬영하는 모습](./images/phone-video.jpg)

예를 들어 전자책 판매자라면 첫 문장은 "전자책이 안 팔릴 때 먼저 볼 것은 표지가 아니라 목차입니다"처럼 잡을 수 있다. 짧지만 어떤 문제를 다루는지 바로 보인다.

## 세 번째 결과물은 상세페이지 문구

상세페이지 문구는 전후 비교가 좋다. 원래 문구 한 문단, 수정 문구 한 문단, 바꾼 이유 한 줄을 붙인다. 이 방식은 작업자의 판단을 보여준다.

![온라인 상품 판매 화면과 작업 책상](./images/ecommerce-desk.jpg)

포트폴리오에 긴 자기소개는 많이 필요하지 않다. 대신 작업 범위를 분명히 적는다. 가능한 작업, 걸리는 시간, 수정 횟수, 필요한 자료를 써두면 된다.

AI 부업은 말로 설명하면 비슷해 보인다. 결과물이 있어야 달라진다. 처음부터 완벽한 포트폴리오를 만들려고 미루기보다 작은 결과물 3개를 먼저 공개하는 편이 빠르다.

## 포트폴리오 글에는 실제 공개 링크가 필요하다

AI 부업 포트폴리오는 이미지 몇 장보다 공개 링크가 중요하다. 특히 바이브 코딩이나 원고 대행을 하려면 결과물을 어디에 올릴지 정해야 한다. GitHub는 개발 포트폴리오의 기본 화면으로 쓰기 좋다. 아래 공개 홈 화면처럼 검색, 저장소, Copilot, 가입 흐름이 보인다.

![GitHub 공개 홈 화면. 코드와 프로젝트를 공개하는 포트폴리오 공간으로 쓸 수 있다.](./images/github-home-screen.jpg)

GitHub가 개발자만 쓰는 곳처럼 보일 수 있지만, 초보 AI 부업 포트폴리오에도 쓸 수 있다. 랜딩페이지 샘플, 계산기 샘플, 문구 전후 비교 문서, README 형식의 작업 기록을 올리면 된다. 중요한 건 "제가 할 수 있습니다"가 아니라 "이 결과물을 공개해두었습니다"다.

## 포트폴리오 3종을 실제 산출물로 바꾼다

처음 만들 포트폴리오는 세 가지면 충분하다고 했다. 이제 그 세 가지를 더 구체적으로 만든다.

| 포트폴리오 | 공개할 파일 | 보여줄 능력 |
|---|---|---|
| 블로그 원고 샘플 | before-after.md | AI 초안을 사람 문장으로 고치는 능력 |
| 숏폼 대본 샘플 | shortform-script.md | 첫 3초 후킹과 CTA 구성 |
| 랜딩페이지 샘플 | index.html 또는 배포 링크 | 바이브 코딩 결과물 검수 능력 |

이 세 가지를 한 페이지에 모으면 된다. Notion 페이지로 만들어도 되고, GitHub README로 만들어도 된다. 개발 작업이 조금 들어간다면 GitHub Pages로 배포하는 편이 좋다. 의뢰인은 파일보다 URL을 누르는 쪽이 편하다.

## v0와 Cursor 결과물을 포트폴리오로 쓰는 방식

바이브 코딩 포트폴리오는 도구 이름을 숨길 필요가 없다. 오히려 어떤 도구를 썼고, 사람이 무엇을 고쳤는지 설명하는 편이 낫다. v0로 초안을 만들고 Cursor로 수정한 뒤 GitHub에 올렸다면, 포트폴리오에는 아래처럼 쓴다.

| 단계 | 설명 |
|---|---|
| 초안 | v0로 전자책 판매 랜딩페이지 구조 생성 |
| 수정 | Cursor로 모바일 여백, 버튼 링크, 문구 수정 |
| 검수 | 360px, 768px, 1440px 화면에서 확인 |
| 배포 | GitHub Pages 또는 Vercel 링크 제공 |
| 한계 | 결제 기능은 외부 링크로 연결, 개인정보 저장 없음 |

이렇게 쓰면 "AI가 다 만든 거 아닌가?"라는 질문을 피할 수 있다. AI가 만든 초안은 누구나 만들 수 있다. 포트폴리오에서 보여줄 것은 초안을 결과물로 다듬는 판단이다.

## 전후 비교가 가장 강한 증거다

포트폴리오에는 완성본만 올리지 않는다. 전후 비교를 넣는다. 예를 들어 블로그 원고라면 아래처럼 짧게 보여준다.

| 구분 | 문장 |
|---|---|
| AI 초안 | 고객 만족을 위해 최선을 다하는 서비스를 제공합니다 |
| 수정본 | 첫 상담에서 예산, 일정, 필요한 자료를 먼저 확인하고 작업 범위를 문서로 남깁니다 |
| 수정 이유 | 빈말을 빼고 실제 진행 절차로 바꿨다 |

숏폼 대본도 같은 방식이다.

| 구분 | 문장 |
|---|---|
| AI 초안 | AI 부업으로 누구나 쉽게 돈을 벌 수 있습니다 |
| 수정본 | Outlier에 가입해도 프로젝트가 비어 있으면 그날은 할 일이 없습니다 |
| 수정 이유 | 과장된 약속을 지우고 실제 리스크를 넣었다 |

이런 비교는 길지 않아도 강하다. 의뢰인은 문장 실력보다 판단 기준을 보고 싶어 한다.

## 포트폴리오 페이지 구성

처음 포트폴리오 페이지는 아래 순서면 충분하다.

1. 닉네임과 한 줄 설명
2. 가능한 작업 3개
3. 샘플 3개
4. 작업 전 필요한 자료
5. 수정 범위
6. 문의 버튼

여기서 자기소개를 길게 쓰지 않는다. "AI를 활용해 다양한 콘텐츠를 제작합니다" 같은 문장은 흔하다. 대신 "AI 초안을 블로그 원고, 숏폼 대본, 랜딩페이지 문구로 고칩니다"처럼 결과물을 말한다.

## 의뢰인이 믿는 포인트

초보 포트폴리오에서 의뢰인이 보는 포인트는 아래다.

| 포인트 | 설명 |
|---|---|
| 공개 링크 | 실제로 눌러볼 수 있는가 |
| 수정 이유 | 왜 그렇게 고쳤는지 말할 수 있는가 |
| 범위 | 어디까지 해주는지 적혀 있는가 |
| 일정 | 하루, 3일, 1주 단위로 납품이 가능한가 |
| 리스크 | 못 하는 작업도 적혀 있는가 |

못 하는 작업을 적는 것도 중요하다. 결제 시스템, 개인정보 저장, 병원·법률 문구 검수처럼 책임이 큰 일은 초반에 피한다. 포트폴리오가 작은 대신 범위가 선명해야 한다.

## 블로그와 포트폴리오를 연결한다

애드센스 블로그를 같이 운영한다면 포트폴리오 글은 내부 링크 허브가 된다. 블로그 글에서 "실제 샘플은 포트폴리오 글에 모아두었다"라고 연결한다. 반대로 포트폴리오 글에서는 관련 글을 연결한다.

- Outlier 플랫폼 비교
- 바이브 코딩 부업
- 블로그 원고 대행
- 숏폼 대본 납품
- 전자책 판매 페이지 문구

이 구조를 만들면 블로그가 단순 기록장이 아니라 작업 증거가 된다. 방문자는 글을 읽고, 샘플을 보고, 문의 버튼을 누른다. 처음부터 대단한 경력이 없어도 된다. 공개된 결과물 3개와 수정 이유가 있으면 출발할 수 있다.

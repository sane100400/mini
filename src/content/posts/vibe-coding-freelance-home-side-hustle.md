바이브 코딩으로 부업을 하겠다고 하면 대부분 "앱을 하나 만들어서 팔면 되나"부터 생각한다. 이 생각이 제일 위험하다. 앱은 작아 보여도 로그인, 결제, 데이터 저장, 오류 처리, 배포, 유지보수가 따라온다. 초보 프리랜서가 집에서 시작하기에는 범위가 너무 쉽게 커진다.

처음 팔아야 하는 것은 앱이 아니라 작은 납품물이다. 랜딩페이지 한 장, 문의 폼 연결, 가격 계산기, FAQ 검색, 전자책 판매 페이지, 스마트스토어 상세페이지용 미니 도구처럼 끝이 보이는 작업이 낫다.

![Cursor 공개 홈 화면. AI 코딩 에이전트가 랜딩페이지를 만드는 예시 화면이 보인다.](./images/cursor-home-screen.jpg)

## 바이브 코딩은 개발을 대신하지 않는다

v0나 Cursor 같은 도구는 초안을 빠르게 만든다. v0는 프롬프트로 UI와 앱 구조를 뽑고, Cursor는 기존 코드 안에서 수정과 리팩터링을 도와준다. 그런데 이 둘이 해주는 일은 납품의 전부가 아니다.

의뢰인이 돈을 내는 지점은 프롬프트가 아니다. 작동하는 링크, 깨지지 않는 모바일 화면, 문의가 도착하는 폼, 수정 범위가 정리된 파일에 돈을 낸다. 도구가 빠르게 만들어준 코드를 사람이 읽고 줄이는 과정이 빠지면, 결과물은 그럴듯한 데모에서 멈춘다.

## 처음 팔기 좋은 5가지

| 상품 | 납품물 | 피해야 할 범위 |
|---|---|---|
| 1페이지 랜딩페이지 | 반응형 HTML, 배포 링크, 문의 버튼 | 회원가입, 결제, 관리자 페이지 |
| 전자책 판매 페이지 | 소개 섹션, 목차, 후기 영역, 구매 버튼 연결 | 자체 결제 시스템 |
| 스마트스토어 보조 계산기 | 옵션별 가격·배송비 계산 UI | 실제 주문 처리 |
| 학원·병원 FAQ 페이지 | 질문 검색, 카테고리, 상담 버튼 | 의료·법률 판단 문구 |
| 블로그용 무료 도구 | 체크리스트, 계산기, 결과 복사 버튼 | 개인정보 저장 |

초반에는 "저도 앱 만들 수 있어요"보다 "이 범위 안에서는 오늘 끝낼 수 있어요"가 더 세다. 작은 납품물은 가격이 낮아도 후기가 생긴다. 후기가 생겨야 더 큰 일을 받을 수 있다.

![v0 공개 홈 화면. 프롬프트 입력창과 랜딩페이지·대시보드 템플릿이 보인다.](./images/v0-home-screen.jpg)

## 작업 순서

나는 바이브 코딩 부업을 아래 순서로 쪼개는 편이 맞다고 본다.

| 단계 | 할 일 | 체크할 것 |
|---|---|---|
| 1. 범위 고정 | 페이지 수, 기능, 수정 횟수 정하기 | "간단히"라는 말을 기능 목록으로 바꾼다 |
| 2. 초안 생성 | v0 또는 Cursor로 UI 초안 만들기 | 한 번에 큰 앱을 시키지 않는다 |
| 3. 코드 정리 | 필요 없는 상태값, 더미 문구, 깨진 링크 제거 | 사람이 읽을 수 있는 구조로 만든다 |
| 4. 모바일 확인 | 360px, 768px, 1440px에서 확인 | 버튼 겹침, 글자 줄바꿈, 이미지 비율 |
| 5. 배포 | GitHub Pages, Vercel, Netlify 중 하나 | 의뢰인이 볼 URL 제공 |
| 6. 인수인계 | 수정 방법, 이미지 교체 위치, 한계 설명 | 유지보수 범위를 문서로 남긴다 |

여기서 돈을 받는 부분은 2번이 아니라 3번부터다. 프롬프트 입력은 누구나 한다. 하지만 깨진 부분을 찾아내고, 의뢰인이 이해할 말로 정리하고, 나중에 수정할 길을 남기는 일은 사람이 맡는다.

## 실제 프롬프트는 짧게 쪼갠다

긴 프롬프트 하나로 끝내려 하면 결과물이 흔해진다. 아래처럼 단계별로 나눠야 덜 망가진다.

| 목적 | 프롬프트 방향 |
|---|---|
| 랜딩페이지 초안 | "전자책 판매용 1페이지 랜딩을 만들어줘. 섹션은 문제, 목차, 샘플, 후기, FAQ, 구매 버튼만 둬." |
| 모바일 정리 | "360px 화면에서 버튼과 제목이 겹치지 않게 여백과 글자 크기를 조정해줘." |
| 문구 교체 | "AI 느낌 나는 과장 문구를 빼고, 전자책 구매자가 확인할 정보 중심으로 바꿔줘." |
| 코드 정리 | "쓰지 않는 컴포넌트와 더미 데이터를 제거하고 파일 구조를 단순하게 만들어줘." |

프롬프트를 잘 쓰는 것보다 더 중요한 것은 결과를 의심하는 습관이다. 링크가 실제로 열리는지, 버튼이 같은 곳으로만 가는지, 모바일에서 표가 터지는지 직접 눌러봐야 한다.

## 의뢰 전에 받아야 할 자료

바이브 코딩 부업에서 작업이 늘어지는 이유는 대부분 자료 부족이다. 시작 전에 최소한 아래 자료를 받아야 한다.

| 자료 | 없으면 생기는 문제 |
|---|---|
| 로고와 브랜드 색 | 임시 디자인이 계속 수정된다 |
| 판매할 상품 설명 | 문구가 추상적으로 흐른다 |
| 연결할 링크 | 버튼이 데모 상태로 남는다 |
| 참고 사이트 2개 | 취향 설명만 길어진다 |
| 수정 우선순위 | 작은 문구 수정에 시간이 새어 나간다 |

자료가 없을 때는 "제가 알아서 예쁘게"라고 받지 않는다. 예시 사이트를 먼저 고르게 하고, 문구는 임시 문구라고 못 박는다. 그래야 수정이 디자인 취향 싸움으로 번지지 않는다.

## 팔면 안 되는 작업도 정해둔다

초보가 피해야 할 작업이 있다.

- 자체 결제 시스템
- 개인정보를 저장하는 회원가입 기능
- 병원·세무·법률처럼 문구 책임이 큰 페이지
- 실시간 예약 시스템
- 운영자가 매일 쓰는 관리자 페이지
- 오류가 나면 매출이 바로 끊기는 쇼핑몰 핵심 기능

이런 일은 돈이 커 보여도 책임이 같이 커진다. 바이브 코딩 도구로 화면은 만들 수 있지만, 보안과 데이터 처리는 다른 문제다. 처음에는 정적 페이지, 외부 폼 연결, 단순 계산기 정도에서 끝내는 편이 오래 간다.

## 포트폴리오 3개만 만든다

처음부터 실제 의뢰를 기다릴 필요는 없다. 가짜 브랜드로 샘플 3개를 만들면 된다.

![GitHub 공개 홈 화면. 바이브 코딩 결과물을 저장소와 배포 링크로 보여줄 때 기준이 된다.](./images/github-home-screen.jpg)

| 샘플 | 넣을 기능 | 보여줄 포인트 |
|---|---|---|
| 전자책 판매 페이지 | 목차, 미리보기, 구매 버튼 | 내가 팔던 전자책과 연결하기 쉽다 |
| 스마트스토어 배송비 계산기 | 옵션 선택, 결과 복사 | 판매자 문제를 직접 건드린다 |
| 학원 상담 랜딩페이지 | FAQ, 상담 링크, 시간표 | 로컬 업종에 제안하기 좋다 |

샘플마다 캡처, 배포 링크, 수정 전후 메모를 붙인다. 완성 링크만 던지면 도구가 만든 것처럼 보인다. 무엇을 고쳤는지 적어야 작업자로 보인다.

## 내 판단

바이브 코딩 부업은 과장해서 보면 금방 실망한다. 앱 하나를 만들어 월수익을 만든다는 말은 듣기 좋지만, 초보에게는 유지보수가 먼저 덮친다.

대신 작게 보면 할 만하다. 전자책 판매 페이지, 문의 랜딩, 계산기, FAQ 같은 작은 납품물은 집에서 만들고 배포까지 끝낸다. 중요한 건 AI가 만든 화면을 그대로 팔지 않는 일이다. 사람이 검수한 작동 링크를 판다.

참고한 공식 페이지: [v0](https://v0.dev/), [Cursor](https://cursor.com/)

## GitHub 링크까지 있어야 외주처럼 보인다

바이브 코딩 결과물을 이미지로만 보여주면 데모처럼 보인다. GitHub 저장소나 배포 링크가 있어야 외주 작업처럼 보인다. 의뢰인은 코드 전체를 읽지 않더라도, 결과물이 어딘가에 정리되어 있다는 사실을 본다.

초보라면 저장소 README에 아래 내용을 넣는다.

| 항목 | 내용 |
|---|---|
| 작업 목적 | 전자책 판매 페이지, 배송비 계산기, 문의 랜딩 중 무엇인지 |
| 사용 도구 | v0, Cursor, Astro, React처럼 실제 쓴 도구 |
| 수정한 부분 | 모바일 여백, 버튼 링크, 문구, 이미지 교체 |
| 배포 링크 | 의뢰인이 바로 눌러볼 URL |
| 제외 범위 | 결제, 회원가입, 개인정보 저장은 포함하지 않음 |

이 표 하나만 있어도 포트폴리오가 훨씬 선명해진다. 바이브 코딩 부업의 핵심은 AI가 만든 초안을 자랑하는 게 아니다. 의뢰인이 볼 수 있는 URL과, 나중에 수정할 수 있는 정리된 파일을 남기는 것이다.

처음에는 저장소를 예쁘게 꾸밀 필요가 없다. README, 스크린샷, 배포 링크, 수정 범위만 있으면 된다. 작은 결과물을 세 개 만들고 같은 형식으로 정리하면, 블로그 글도 포트폴리오도 동시에 쌓인다.

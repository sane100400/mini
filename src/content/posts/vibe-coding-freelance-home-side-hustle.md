부업으로 바이브 코딩을 떠올리면 제일 먼저 앱 하나를 만들고 싶어진다. v0에 말하면 화면이 나오고, Cursor로 고치면 코드도 움직인다. 다만 외주로 팔려면 앱 전체가 아니라 바로 보여줄 수 있는 작은 작업부터 골라야 한다.

처음부터 앱 전체를 맡기보다 1페이지 랜딩페이지, 계산기, 문의 폼, README 정리처럼 결과가 분명한 작업부터 제안하는 편이 낫다. 설명하기도 쉽고 포트폴리오로 남기기도 좋다. 로그인, 결제, 개인정보 저장, 관리자 기능, 데이터베이스처럼 책임이 큰 기능은 경험이 쌓인 뒤에 넓히면 된다.

| 먼저 제안하기 좋은 것 | 나중에 넓힐 것 |
| --- | --- |
| 1페이지 랜딩페이지 | 결제 기능 |
| 계산기나 견적 폼 | 회원가입·로그인 |
| 문의 폼 화면 | 개인정보 저장 |
| 가격표·FAQ 섹션 | 관리자 페이지 |
| GitHub README 정리 | 예약 시스템 |

## Cursor는 고쳐보는 데 편했다

상단 대표 이미지는 Cursor 공개 홈 화면을 직접 캡처했다. AI 코딩 에이전트가 랜딩페이지를 만드는 예시가 나온다. 문장으로 요청하면 코드가 바뀌고 화면도 따라 바뀐다. 그래서 처음에는 쉬워 보인다.

코드는 도구가 빨리 만들어줘도 마무리는 사람이 해야 한다. 문구가 맞는지, 모바일에서 깨지지 않는지, 버튼이 실제 링크로 이어지는지, 이미지가 무겁지 않은지, 배포 경로가 맞는지 봐야 작업 품질이 선명해진다.

## v0는 첫 화면 초안을 만들 때 편했다

v0 화면도 같이 봤다. v0 공개 홈에는 프롬프트 입력창과 랜딩페이지·대시보드 템플릿이 있다. 완성품을 한 번에 뽑는 도구라기보다 첫 화면 초안을 빨리 만드는 쪽에 더 잘 맞았다.

| 도구 | 맡기기 좋은 일 | 사람이 마무리할 일 |
| --- | --- | --- |
| v0 | 랜딩페이지, 카드 UI, 입력 폼 초안 | 실제 프로젝트에 맞게 정리 |
| Cursor | 코드 수정, 문구 교체, 구조 정리 | 결과 검수와 링크 확인 |
| GitHub | 저장소, README, 배포 기록 | 작업 설명과 배포 링크 정리 |
| GitHub Pages | 정적 페이지 배포 | 경로와 이미지 링크 확인 필요 |

## 처음 제안하기 좋은 작업

처음 제안하기 좋은 작업은 다섯 가지 정도로 좁혔다. 중요한 건 "멋진 앱"이 아니라 끝까지 확인할 수 있는 범위다.

| 작업 | 어울리는 의뢰 | 끝난 것으로 볼 기준 |
| --- | --- | --- |
| 1페이지 랜딩페이지 | 전자책, 온라인 클래스, 작은 서비스 소개 | 섹션 5개, 모바일 대응, CTA 링크 |
| 계산기나 견적 폼 | 가격 추정, 점수 계산, 간단한 진단 | 입력값, 결과값, 오류 문구 |
| 문의 폼과 FAQ 섹션 | 상담 유도 페이지 | 폼 화면, FAQ 5개, 연결 링크 |
| 기존 페이지 문구·레이아웃 수정 | 이미 사이트가 있는 의뢰인 | 수정 전후 비교 |
| GitHub README와 배포 안내 | 포트폴리오 정리 | README, 배포 링크, 사용 도구 |

작게 시작한다는 말은 단가를 낮게만 받으라는 뜻이 아니다. 범위를 명확히 하라는 뜻이다. "랜딩페이지 만들어드립니다"보다 "전자책 판매용 1페이지 랜딩페이지, 섹션 5개, 모바일 대응, 배포 링크 포함"처럼 적어야 한다.

## 작업 순서를 정해두면 대화가 빨라진다

바이브 코딩 작업은 순서를 정해두면 훨씬 편하다. 화면을 먼저 나누고, 문구와 링크를 받은 뒤 v0와 Cursor로 초안을 다듬으면 어디까지 해줄지 흔들리지 않는다.

| 순서 | 할 일 |
| --- | --- |
| 1 | 의뢰인의 목적 확인 |
| 2 | 반드시 들어갈 문구와 이미지 받기 |
| 3 | 화면 구조 나누기 |
| 4 | v0 같은 도구로 초안 만들기 |
| 5 | Cursor에서 문구, 링크, 반응형 수정 |
| 6 | GitHub와 배포 링크 정리 |
| 7 | 모바일, 버튼, 이미지 확인 |

프롬프트도 짧게 쪼갠다. 한 번에 "멋진 랜딩페이지 만들어줘"라고 하면 결과가 두루뭉술해진다. "전자책 판매용 첫 화면을 만들고 제목은 AI 부업 30일 운영표로 해줘", "가격 섹션은 카드 2개로 나눠줘", "모바일에서 버튼이 아래로 줄바꿈되게 해줘"처럼 단계별로 요청한다.

## 시작 전에 받을 자료

자료를 먼저 받으면 AI가 마음대로 채우는 문구를 줄일 수 있다. 상품명, 가격, 이미지, 연결 링크가 있으면 작은 랜딩페이지도 훨씬 빨리 정리된다.

| 받아야 할 자료 | 이유 |
| --- | --- |
| 상품명과 한 줄 소개 | 첫 화면 문구 |
| 주요 섹션 | 페이지 구조 |
| 가격과 혜택 | 판매 과정 |
| 이미지와 로고 | 디자인 맞춤 |
| 참고 사이트 | 원하는 분위기 확인 |
| 연결할 링크 | 버튼 검수 |
| 문의 방법 | CTA 설정 |

나중에 맡을 일도 미리 선을 그어둔다. 결제 정보를 직접 처리하는 기능, 개인정보를 저장하는 기능, 법률·의료처럼 검수가 필요한 기능, 보안 책임이 큰 기능은 경험과 협업 체계가 생긴 뒤에 다루는 편이 좋다. 초반에는 외부 결제 링크나 구글폼·타입폼 같은 기존 도구를 연결하는 정도로 시작할 수 있다.

## 포트폴리오는 3개만 있어도 말이 통한다

바이브 코딩 포트폴리오는 3개면 이야기가 된다. 전자책 판매 랜딩페이지, 간단한 가격 계산기, 개인 서비스 소개 페이지 정도다. 각 샘플에는 문제, 만든 것, 사용 도구, 배포 링크, GitHub 링크를 적는다.

| 포트폴리오 | 보여줄 것 |
| --- | --- |
| 전자책 랜딩페이지 | 섹션 구성, CTA, 모바일 화면 |
| 가격 계산기 | 입력값, 결과값, 에러 처리 |
| 개인 서비스 소개 페이지 | 프로필, 작업 범위, 문의 버튼 |

README도 같이 쓴다. 무엇을 만들었는지, 어떤 도구를 썼는지, 어디까지가 샘플인지, 배포 링크가 어디인지 적어야 한다. 의뢰인은 코드를 다 읽지 않아도 공개 링크가 있다는 사실에서 신뢰를 느낀다.

## 마무리

바이브 코딩 부업은 작은 화면, 작은 기능, 작은 수정부터 시작할 때 설명하기 쉽다. AI 도구는 초안을 빨리 만들고, 사람은 의뢰인에게 넘겨도 되는 상태로 다듬는다. 이렇게 나눠두면 외주 상품으로도 블로그 글감으로도 쓰기 좋다.

애드센스 블로그 글감으로도 바이브 코딩은 괜찮다. v0로 랜딩페이지 샘플 만들기, Cursor로 문구 고치기, GitHub Pages로 포트폴리오 배포하기, 작은 외주 범위를 정하는 법처럼 글로 나눠 쓰기 쉽다.

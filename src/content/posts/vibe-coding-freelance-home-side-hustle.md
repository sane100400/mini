바이브 코딩을 부업으로 생각하면 가장 먼저 앱 하나를 만들고 싶어진다. AI에게 설명하면 화면이 나오고, 코드를 고치면 작동하는 것처럼 보이기 때문이다. 하지만 프리랜서 일로 팔 때는 앱 하나보다 작은 납품물이 먼저다. 작은 결과물을 안정적으로 만들고, 어디까지 책임질지 정해야 오래 간다.

바이브 코딩은 개발을 대신하지 않는다. 개발자가 아니어도 시작할 수 있는 부분은 있지만, 검수와 책임까지 사라지는 것은 아니다. 로그인, 결제, 개인정보, 관리자 기능, 데이터 저장처럼 민감한 기능은 생각보다 복잡하다. 초보가 처음부터 이런 작업을 맡으면 수정과 장애 대응에서 막힌다. 그래서 처음에는 랜딩페이지, 계산기, 문의 폼, 가격표, README 정리처럼 작은 결과물부터 보는 편이 낫다.

## 도구 화면을 먼저 본다

아래는 Cursor 공개 홈 화면이다. AI 코딩 에이전트가 랜딩페이지를 만드는 예시 화면이 보인다. 이 화면을 보면 바이브 코딩이 어떤 식으로 느껴지는지 알 수 있다. 문장으로 요청하고, 코드가 바뀌고, 화면이 만들어진다. 그래서 쉬워 보인다.

![Cursor 공개 홈 화면. AI 코딩 에이전트가 랜딩페이지를 만드는 예시 화면이 보인다.](./images/cursor-home-screen.jpg)

하지만 도구가 코드를 만들어준다고 해서 결과물이 자동으로 납품 가능한 상태가 되는 것은 아니다. 문구가 맞는지, 모바일에서 깨지지 않는지, 버튼이 실제 링크로 이어지는지, 이미지가 무겁지 않은지, 배포 경로가 맞는지 확인해야 한다. 의뢰인은 "AI가 만들어준 코드"를 사는 것이 아니라 "쓸 수 있는 화면"을 산다.

v0 화면도 같이 볼 필요가 있다. 아래는 v0 공개 홈 화면이다. 프롬프트 입력창과 랜딩페이지·대시보드 템플릿이 보인다. v0는 화면 초안을 빠르게 만드는 데 좋다. 특히 랜딩페이지, 대시보드, 카드형 UI, 입력 폼 같은 작업은 초안을 빨리 볼 수 있다.

![v0 공개 홈 화면. 프롬프트 입력창과 랜딩페이지·대시보드 템플릿이 보인다.](./images/v0-home-screen.jpg)

v0와 Cursor를 같이 쓰면 흐름이 생긴다. v0로 화면 초안을 만들고, Cursor로 프로젝트 안에서 문구와 구조를 고치고, GitHub에 올리고, 배포 링크로 보여준다. 이 흐름은 작은 납품물에 잘 맞는다. 반대로 복잡한 서비스 전체를 맡기에는 검수할 것이 너무 많다.

## 처음 팔기 좋은 것은 작다

처음 팔기 좋은 납품물은 다섯 가지 정도다. 첫째, 1페이지 랜딩페이지다. 전자책, 온라인 클래스, 작은 서비스 소개에 맞다. 둘째, 계산기나 견적 폼이다. 입력값을 넣으면 대략적인 결과를 보여주는 정도다. 셋째, 문의 폼과 FAQ 섹션이다. 실제 메일 연동까지 포함할지 여부는 따로 정해야 한다. 넷째, 기존 페이지 문구와 레이아웃 수정이다. 다섯째, GitHub README와 배포 안내 정리다.

이 작업들은 작지만 포트폴리오로 보여주기 좋다. 의뢰인도 결과를 확인하기 쉽다. 버튼을 누르면 이동하는지, 모바일에서 잘 보이는지, 문구가 맞는지 바로 볼 수 있다. 반대로 회원가입, 결제, 예약 시스템, 관리자 페이지, 데이터베이스 설계처럼 민감한 기능은 초반 상품에서 빼는 편이 낫다.

작게 시작한다는 말은 단가를 낮게만 받으라는 뜻이 아니다. 범위를 명확히 하라는 뜻이다. "랜딩페이지 만들어드립니다"보다 "전자책 판매용 1페이지 랜딩페이지, 섹션 5개, 모바일 대응, 배포 링크 포함"처럼 적어야 한다. 수정 횟수와 포함되지 않는 기능도 같이 적는다.

## 작업 순서는 고정한다

바이브 코딩 작업은 순서를 정해두지 않으면 AI와 대화하다가 시간이 흘러간다. 먼저 의뢰인의 목적을 확인한다. 어떤 상품이나 서비스를 소개할지, 방문자가 어떤 행동을 해야 하는지, 반드시 들어갈 문구와 이미지는 무엇인지 받는다. 그다음 화면 구조를 짠다. 히어로, 문제 제기, 기능, 가격, FAQ, CTA처럼 섹션을 나눈다.

구조가 잡히면 v0나 비슷한 도구로 초안을 만든다. 초안이 마음에 들어도 바로 납품하지 않는다. Cursor 같은 도구에서 프로젝트에 맞게 정리하고, 문구를 실제 자료에 맞춘다. 이미지 경로, 링크, 버튼, 반응형 레이아웃을 확인한다. 마지막으로 GitHub와 배포 링크를 준비한다.

프롬프트도 짧게 쪼개야 한다. 한 번에 "멋진 랜딩페이지 만들어줘"라고 하면 결과가 넓게 나온다. "전자책 판매용 첫 화면을 만들고, 제목은 AI 부업 30일 운영표로 해줘", "가격 섹션은 카드 2개로 나눠줘", "모바일에서 버튼이 아래로 줄바꿈되게 해줘"처럼 단계별로 요청한다. 짧은 프롬프트가 수정하기 쉽다.

## 의뢰 전에 받을 자료

바이브 코딩 납품에서도 자료가 중요하다. 상품명, 한 줄 소개, 주요 섹션, 가격, 이미지, 로고, 참고 사이트, 연결할 링크, 문의 방법을 받아야 한다. 자료가 없으면 AI가 임의로 문구를 채운다. 그 문구가 그럴듯해 보여도 실제 의뢰인의 사업과 맞지 않을 수 있다.

작업 전에 물어볼 질문도 정해둔다. 이 페이지의 목적은 판매인지 문의인지, 방문자가 마지막에 눌러야 할 버튼은 무엇인지, 모바일 방문자가 많은지, 배포 후 직접 수정할 계획이 있는지 확인한다. 이 답변에 따라 구조가 달라진다. 판매 페이지라면 가격과 혜택이 중요하고, 문의 페이지라면 신뢰 요소와 폼이 중요하다.

팔면 안 되는 작업도 미리 정한다. 결제 정보를 직접 처리하는 기능, 개인정보를 저장하는 기능, 법률·의료처럼 검수가 필요한 기능, 보안 책임이 큰 기능은 초반에는 받지 않는다. 대신 외부 결제 링크로 이동하거나, 구글폼·타입폼 같은 기존 도구를 연결하는 정도로 범위를 줄일 수 있다. 안전한 범위 안에서만 일해야 한다.

## 포트폴리오는 3개면 시작한다

바이브 코딩 포트폴리오는 3개면 시작할 수 있다. 전자책 판매 랜딩페이지, 간단한 가격 계산기, 개인 서비스 소개 페이지 정도다. 각 포트폴리오에는 문제, 만든 것, 사용 도구, 배포 링크, GitHub 링크를 적는다. 화면만 보여주면 부족하다. 어떤 요청을 받아 어떤 결과로 바꿨는지 설명해야 한다.

아래는 GitHub 공개 홈 화면이다. 바이브 코딩 결과물을 저장소와 배포 링크로 보여줄 때 기준이 된다. GitHub는 코드만 올리는 곳이 아니라 작업 과정을 남기는 공간으로 쓸 수 있다. README에 프로젝트 설명, 사용 도구, 수정 범위, 배포 링크를 적으면 의뢰인이 확인하기 쉽다.

![GitHub 공개 홈 화면. 바이브 코딩 결과물을 저장소와 배포 링크로 보여줄 때 기준이 된다.](./images/github-home-screen.jpg)

GitHub 링크가 있으면 외주처럼 보이는 이유가 있다. 결과물이 파일 하나로 끝나지 않고, 저장소와 배포 링크로 남기 때문이다. 의뢰인은 완성 화면을 보고, 필요하면 코드와 설명도 확인할 수 있다. 물론 모든 의뢰인이 코드를 읽지는 않는다. 그래도 공개 링크가 있다는 사실 자체가 신뢰를 만든다.

## 내 판단

바이브 코딩 부업은 가능성이 있다. 하지만 초보에게 맞는 방식은 앱 전체 개발이 아니다. 작은 화면, 작은 기능, 작은 수정부터 시작해야 한다. AI 도구는 초안을 빨리 만들지만, 납품 가능한 결과물로 만드는 과정은 사람이 책임져야 한다. 버튼, 링크, 모바일, 문구, 이미지, 배포 경로를 확인하지 않으면 돈을 받기 어렵다.

애드센스 블로그 글감으로도 바이브 코딩은 좋다. "v0로 랜딩페이지 샘플 만들기", "Cursor로 문구 수정하는 흐름", "GitHub Pages로 포트폴리오 배포하기", "바이브 코딩 외주에서 받지 말아야 할 작업" 같은 글을 계속 만들 수 있다. 실제 화면 캡처와 샘플 링크가 있으면 글의 신뢰도도 올라간다.

결론은 단순하다. 바이브 코딩을 부업으로 보려면 앱 하나를 약속하지 않는다. 작게 만들고, 공개 링크로 보여주고, 수정 범위를 좁게 적는다. v0와 Cursor는 좋은 도구지만, 의뢰인이 사는 것은 도구 사용 능력이 아니라 쓸 수 있는 결과물이다. 그 결과물을 작게라도 끝까지 만드는 것이 첫 단계다.

처음 만든 AI 부업 블로그 운영표는 단순했다. 며칠마다 글을 올릴지, 어떤 카테고리를 채울지 정도만 적었다. 하지만 애드센스 블로그는 발행량만으로 버티기 어렵다. 검색자가 실제로 막히는 질문을 어떤 순서로 쌓는지가 더 중요하다.

30일 운영표의 기준은 글 개수가 아니라 정보의 깊이다. 첫 주에는 플랫폼과 도구 화면을 확인하고, 둘째 주에는 팔 수 있는 작은 납품물을 정리한다. 셋째 주에는 포트폴리오와 판매 흐름으로 연결하고, 넷째 주에는 Search Console과 글별 반응을 보고 제목을 고친다.

| 주차 | 핵심 목표 | 쓰는 글의 역할 |
| --- | --- | --- |
| 1주차 | 플랫폼과 도구 확인 | 검색 유입 입구 만들기 |
| 2주차 | 작은 납품물 정리 | 프리랜서 상품화 |
| 3주차 | 포트폴리오와 판매 연결 | 신뢰와 전환 만들기 |
| 4주차 | 지표와 수정 | 다음 달 방향 결정 |

## 1주차는 실제 화면부터 확인한다

AI 부업을 검색하는 사람은 먼저 이름을 본다. Outlier, DataAnnotation, Alignerr, Mindrift 같은 플랫폼 이름을 어디선가 듣고 들어온다. 블로그가 해야 할 일은 추천 목록을 반복하는 것이 아니라, 가입 전에 무엇을 확인해야 하는지 정리하는 것이다.

| 1주차 글감 | 넣을 자료 | 목적 |
| --- | --- | --- |
| AI 데이터 라벨링 부업 팁, Outlier·DataAnnotation 한국어 작업 조건 확인 | Outlier, DataAnnotation, Alignerr 화면 | 검색 유입 |
| 애드센스 승인 전 체크리스트 | AdSense 화면, sitemap 점검표 | 운영 기준 |
| 바이브 코딩 도구 확인 | Cursor, v0 화면 | 다음 주 납품물 글로 연결 |

첫 주에 전자책, 숏폼, 스마트스토어, 바이브 코딩을 모두 깊게 쓰려고 하면 방향이 흐려진다. 먼저 이 블로그가 "실제 화면과 작은 결과물로 AI 부업을 검토한다"는 기준을 보여줘야 한다.

## 2주차는 작은 납품물을 정한다

둘째 주에는 플랫폼에서 벗어나 실제로 팔 수 있는 작은 결과물을 본다. 바이브 코딩이라면 앱 하나가 아니라 랜딩페이지 수정, 계산기, 문의 폼, README 정리 같은 작은 납품물이다. 숏폼 제작이라면 완성 영상이 아니라 대본, 자막 문장, 썸네일 문구부터 시작한다.

| 부업 후보 | 처음 팔기 좋은 납품물 | 나중에 넓힐 것 |
| --- | --- | --- |
| 바이브 코딩 | 1페이지 랜딩페이지, 계산기, FAQ 섹션 | 결제, 로그인, 개인정보 저장 |
| 숏폼 제작 | 20초 대본, 자막 문장, 썸네일 문구 | 전체 채널 운영 |
| 블로그 원고 | 제목, 목차, 본문 초안, 수정 1회 | 장문 원고와 SEO 옵션 |
| 상세페이지 문구 | 고객 질문표, 문구 전후 비교 | 디자인 전체 리뉴얼 |

이 주간의 글은 "무엇을 팔 것인가"에 답해야 한다. 거창한 성공 사례보다 초보가 집에서 만들고, 포트폴리오로 보여주고, 의뢰인에게 범위를 설명할 준비가 됐는지가 중요하다.

## 3주차는 포트폴리오로 연결한다

셋째 주에는 가능하다는 말을 결과물로 바꾼다. AI 부업 포트폴리오는 화려할 필요가 없다. 블로그 글, 숏폼 대본, 상세페이지 문구처럼 작지만 완성된 결과물 3개면 시작한다.

| 포트폴리오 결과물 | 보여줘야 할 것 | 연결 글감 |
| --- | --- | --- |
| 블로그 글 | 제목을 좁힌 과정, 공식 화면, 결론 | 글감 찾기, 원고 대행 |
| 숏폼 대본 | 후킹, 본문, CTA, 자막 문장 | 숏폼 대본 외주 |
| 상세페이지 문구 | 고객 질문, 수정 전후 비교 | 스마트스토어 문구 개선 |
| 랜딩페이지 | 배포 링크, GitHub, README | 바이브 코딩 부업 |

이 주간에는 전자책과 템플릿 판매 흐름도 본다. 전자책을 만든다면 업데이트 로그와 번들 구성을 글로 남기고, Notion·Canva 템플릿을 만든다면 사용 장면과 예시 데이터를 정리한다.

## 4주차는 숫자보다 질문을 본다

넷째 주에야 지표를 본다. 조회수, 클릭, 체류 시간, 검색 노출은 중요하다. 다만 첫 3주 동안 글의 기준이 없었다면 숫자를 봐도 해석이 어렵다.

| 볼 지표 | 질문 |
| --- | --- |
| 검색 노출 | 어떤 질문으로 들어왔나 |
| 클릭률 | 제목이 너무 넓거나 흐린가 |
| 체류 시간 | 표와 이미지가 글을 붙잡고 있나 |
| 내부 링크 | 다음 글로 넘어갈 이유가 있나 |
| 전환 | 전자책, 문의, 포트폴리오 클릭이 있는가 |

넷째 주에는 새 글보다 수정이 더 중요할 때가 많다. 제목이 넓은 글은 좁힌다. 본문 앞뒤가 따로 노는 글은 흐름을 다시 잡는다. 장식 이미지뿐인 글에는 실제 서비스 화면이나 직접 만든 샘플을 넣는다.

## 하루 2시간 운영표

현실적으로 이 블로그에 하루 종일 매달리기는 어렵다. 그래서 하루 2시간 기준으로 운영표를 잡았다.

| 시간 | 할 일 | 결과물 |
| --- | --- | --- |
| 30분 | 공식 페이지, 공개 마켓, 이전 글 확인 | 참고자료와 글 질문 |
| 40분 | 초안 작성 | 제목, 목차, 본문 초안 |
| 30분 | 이미지와 근거 확인 | 캡처, 표, 샘플 |
| 20분 | 제목, 내부 링크, 모바일 확인 | 발행 전 체크 |

자료 확인 없이 초안을 쓰면 글이 일반론으로 흐른다. 이미지 확인 없이 발행하면 실제 사례가 빠진다. 내부 링크를 마지막에 보지 않으면 글들이 서로 따로 논다.

## 30일 뒤 남아야 하는 것

| 남길 자산 | 완료 기준 |
| --- | --- |
| 플랫폼 비교 글 | 실제 화면과 가입 전 기준 포함 |
| 애드센스 운영 글 | sitemap, 정책 페이지, RPM 기준 포함 |
| 프리랜서 납품물 글 | 작업 범위와 수정 기준 포함 |
| 포트폴리오 샘플 | 공개 링크 또는 전후 비교 포함 |
| 다음 달 글감표 | 검색어와 막힌 질문 기준으로 정리 |

AI 부업 블로그는 빠르게 많이 쓰는 것만으로는 부족하다. 검색자가 막힌 지점을 보고, 그 지점을 실제 화면으로 확인하고, 집에서 만드는 작은 산출물로 바꾸는 흐름이 필요하다. 30일 운영표는 그 흐름을 잃지 않기 위한 기준표다.

애드센스 블로그를 시작할 때 가장 먼저 묻는 것은 보통 글 개수다. 승인 전에 몇 개를 써야 하는지, AI로 쓴 글도 괜찮은지, 카테고리는 몇 개가 필요한지부터 궁금해진다. 그런데 실제로 점검해보면 순서는 조금 다르다. 글 개수보다 먼저 확인해야 할 것은 이 사이트가 어떤 문제를 꾸준히 다루는지다.

이 블로그는 범위를 "한국인이 집에서 직접 확인 가능한 AI 부업"으로 좁혔다. 애드센스, AI 트레이닝 플랫폼, 바이브 코딩, 전자책, 숏폼 대본, 상세페이지 문구처럼 직접 화면을 보고 판단하는 주제만 남겼다. Google은 검색 순위를 조작하기 위한 글보다 사람에게 도움이 되는 콘텐츠를 만들라고 안내한다. 그래서 승인 전 점검도 "글을 많이 썼는가"보다 "검색자가 판단할 근거가 있는가"로 봐야 한다.

| 먼저 볼 항목 | 통과 기준 | 부족할 때 고칠 것 |
| --- | --- | --- |
| 주제 범위 | 한 문장으로 설명됨 | 너무 넓은 부업 추천글 줄이기 |
| 글의 역할 | 검색 유입, 비교, 포트폴리오, 수익 연결이 나뉨 | 비슷한 글 제목 합치기 |
| 기본 페이지 | 소개, 개인정보처리방침, RSS, sitemap, robots.txt 확인 | 누락 페이지 먼저 만들기 |
| 실제 근거 | 공식 화면, 마켓 화면, 직접 만든 샘플 포함 | 장식 이미지 대신 실제 화면 캡처로 보강 |
| 내부 링크 | 다음 글로 넘어갈 이유가 있음 | 관련 글 묶음 만들기 |

## 1. 블로그 주제를 한 문장으로 좁힌다

"AI로 돈 버는 법"은 너무 넓다. 이런 제목은 쓰기는 쉽지만 본문이 추천 목록으로 흐르기 쉽다. 이 블로그는 아래처럼 주제를 쪼갰다.

| 카테고리 | 다룰 질문 | 넣을 근거 |
| --- | --- | --- |
| 애드센스 | 승인 전에 어떤 글 구조가 필요한가 | AdSense 화면, sitemap, 정책 페이지 |
| AI 트레이닝 플랫폼 | Outlier 같은 플랫폼을 가입 전에 어떻게 볼까 | 공개 기회 화면, 심사 기준표 |
| 바이브 코딩 | 앱 전체 말고 어떤 납품물이 안전한가 | Cursor, v0, GitHub 화면 |
| 숏폼 대본 | 완성 영상 전 어떤 산출물을 팔 수 있나 | CapCut, YouTube Shorts 화면 |
| 전자책 | 판매 후 어떻게 업데이트할까 | Gumroad, 크몽, 업데이트 로그 |

주제가 좁으면 글감이 줄어드는 것처럼 보이지만 실제로는 반대다. "AI 부업 추천"보다 "Outlier 공개 기회가 비어 있을 때 지원해도 되는가", "v0로 만든 랜딩페이지를 외주 샘플로 보여줘도 되는가"가 더 쓰기 쉽다. 검색자도 넓은 주장보다 자기 상황에 가까운 질문을 클릭한다.

## 2. 글은 실험 기록처럼 보여야 한다

애드센스용 글은 문장만 매끈해서는 부족하다. 검색자가 "이 글은 실제로 확인하고 썼다"고 느낄 근거가 있어야 한다. Google의 유용한 콘텐츠 가이드도 독자를 먼저 생각하는 콘텐츠를 강조한다. 우리 블로그에서는 글마다 아래 요소 중 하나 이상을 넣는다.

| 글 안에 넣을 것 | 예시 | 역할 |
| --- | --- | --- |
| 공식 화면 | AdSense, Outlier, CapCut | 서비스 기준 확인 |
| 공개 마켓 화면 | 크몽, Gumroad, Notion Templates | 수요와 판매 흐름 확인 |
| 직접 만든 샘플 | 대본, 상세페이지 문구, 랜딩페이지 | 포트폴리오 역할 |
| 체크리스트 | 신청 전 점검표, 자료 요청표 | 독자가 바로 따라 할 기준 |

상단 대표 이미지는 Google AdSense 시작 페이지를 직접 캡처한 화면이다. 애드센스 글에는 AdSense 화면이, 숏폼 글에는 CapCut 화면이 들어가야 정보 글로 보이는 힘이 생긴다. 이미지는 장식이 아니라 근거로 써야 한다.

## 3. sitemap과 robots.txt를 신청 전에 확인한다

승인 신청 전에는 검색엔진이 사이트 구조를 찾는지도 본다. 이 Astro 블로그는 `src/pages/rss.xml.ts`, `src/pages/robots.txt.ts`, sitemap 설정을 갖고 있다. Google Search Central은 sitemap을 만들어 Google에 제공하고, robots.txt에는 sitemap 위치를 적을 수 있다고 안내한다.

| 확인할 파일 | 확인할 내용 |
| --- | --- |
| `/rss.xml` | 최신 글이 피드에 들어가는가 |
| `/robots.txt` | 크롤러 차단이 과하지 않은가 |
| `/sitemap-index.xml` 또는 sitemap | 글 URL이 빠지지 않는가 |
| canonical URL | 같은 글이 중복 주소로 보이지 않는가 |

robots.txt는 비공개 파일을 숨기는 보안 장치가 아니다. Google 문서도 robots.txt가 크롤러 트래픽 관리용이며, 보안을 위해서는 다른 차단 방식이 필요하다고 설명한다. 블로그에서는 검색에 보여도 되는 공개 글만 발행한다는 전제가 먼저다.

## 4. 정책 페이지와 소개 페이지를 먼저 정리한다

애드센스 블로그는 광고가 붙는 사이트다. 방문자에게 누가 쓰는 글인지, 개인정보와 광고 쿠키를 어떻게 다루는지, 사이트 목적이 무엇인지 보여야 한다.

| 페이지 | 최소 내용 |
| --- | --- |
| 소개 | 블로그 주제, 운영자 관점, 다루지 않는 주제 |
| 개인정보처리방침 | 광고, 분석, 쿠키, 문의 정보 처리 |
| 문의 또는 연락 방법 | 외주 문의나 오류 제보 경로 |
| 카테고리 페이지 | 같은 주제 글을 이어 읽는 구조 |

정책 페이지가 있다고 승인되는 것은 아니다. 다만 없으면 신뢰가 약해진다. 글의 품질과 기본 페이지를 같이 봐야 한다.

## 5. 첫 30개 글을 역할별로 나눈다

첫 글 30개를 숫자로만 채우면 비슷한 글이 늘어난다. 역할을 나누면 같은 주제라도 겹치지 않는다.

| 글의 역할 | 예시 글감 |
| --- | --- |
| 검색 유입 | Outlier·DataAnnotation 가입 전 확인할 것 |
| 비교 판단 | AI 트레이닝 플랫폼별 심사·일감·언어 조건 |
| 실무 샘플 | AI 부업 포트폴리오 3개 만드는 법 |
| 수익 연결 | 전자책 업데이트 로그와 번들 구성 |
| 운영 기록 | AI 부업 블로그 30일 운영표 |

승인 전 글 개수가 10개인지 30개인지는 정답이 없다. 대신 각 글이 서로 다른 질문에 답해야 한다. 제목만 다르고 본문이 같은 글은 사이트를 얇아 보이게 한다.

## 6. 모바일에서 표와 이미지가 깨지지 않는지 본다

애드센스 블로그는 모바일 유입이 많은 편이다. 표가 너무 넓거나 이미지가 무겁거나 광고가 본문을 가리면 읽기 경험이 나빠진다. Google의 페이지 경험 문서는 한두 지표만 보지 말고 전체 경험을 봐야 한다고 설명한다.

| 모바일 점검 | 볼 것 |
| --- | --- |
| 첫 화면 | 제목, 대표 이미지, 첫 문단이 자연스럽게 보이는가 |
| 표 | 가로 스크롤이나 줄바꿈이 과하게 생기지 않는가 |
| 이미지 | 실제 화면 내용이 읽히는가 |
| 내부 링크 | 다음 글로 이동하기 쉬운가 |
| 광고 위치 | 본문 클릭을 방해하지 않는가 |

## 7. 신청은 마지막에 한다

애드센스 신청은 준비의 시작이 아니라 점검의 마지막 단계로 둔다. 신청 전에는 홈 화면에서 최신 글 3개가 블로그 방향을 보여주는지, 글마다 실제 화면이나 표가 들어갔는지, 제목과 본문이 같은 질문을 다루는지 확인한다.

마지막 체크리스트는 아래처럼 잡았다.

| 체크 항목 | 상태 |
| --- | --- |
| 블로그 주제가 한 문장으로 설명되는가 | 한국인이 집에서 확인하는 AI 부업 |
| 글마다 하나의 질문과 결론이 있는가 | 각 글 제목에서 질문을 좁힘 |
| 공식 화면이나 직접 만든 샘플이 있는가 | AdSense, Outlier, CapCut, Cursor 등 캡처 사용 |
| 기본 페이지와 sitemap이 있는가 | 소개, privacy, RSS, robots.txt 확인 |
| 다음 글로 이어지는 내부 링크가 있는가 | 애드센스, 포트폴리오, 전자책 글을 연결 |

애드센스 블로그를 시작한다는 말은 광고 코드를 붙이겠다는 뜻만은 아니다. 특정한 문제를 꾸준히 다루는 사이트를 만든다는 뜻에 가깝다. 이 블로그는 집에서 확인 가능한 AI 부업을 실제 화면과 작은 결과물로 검토하는 방향으로 운영한다.
